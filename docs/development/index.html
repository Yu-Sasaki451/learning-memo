CakePHP 2.0 + VS Code 開発手順

バグ修正や機能追加を始める時に、何を確認し、どの順番で作業するかの手順です。
CentOS環境そのものの説明は、別ページに分けています。

先に見るページ

VS Codeで開いているファイルやターミナルのコマンドが、自分のPCではなくCentOSサーバー側で動くことがある。
その前提が曖昧な時は、先にCentOSの基本ページを見る。

先に確認: CentOSとは何か、通常の開発と何が違うか

公式で確認する情報

CakePHP 2.0は古いので、最新CakePHPの情報をそのまま使わない。
調べる時はCakePHP 2.xの公式ドキュメントを見る。

開発に入る前の確認

作業前に、今いる場所、ブランチ、変更差分、PHPのバージョンを確認する。
ここを飛ばすと、違うブランチや古いコードに対して作業する危険がある。 

pwd
git status
git branch
php -v
pwd 今ターミナルが開いている場所を確認する。CakePHPのプロジェクト直下なのか、appの中なのかを見る。
git status 今のブランチと変更中のファイルを見る。知らない変更がある時は、作業前に確認する。
git branch ブランチ一覧を見る。先頭に * が付いている行が今いるブランチ。
php -v PHPのバージョンを見る。CakePHP 2.0は古いので、環境差で動きが変わることがある。

作業ブランチを作る

mainやdevelopを直接触らず、作業用ブランチを作る。
会社の基準ブランチがdevelopならdevelop、mainならmainから作る。 

git switch develop
git pull
git switch -c fix/ticket-123-user-edit-error
git switch develop 基準ブランチへ移動する。会社がmain運用なら main に変える。
git pull リモートの最新コードを取り込む。実行前に git status で作業中の変更がないか見る。
git switch -c 新しい作業ブランチを作って、そのブランチへ移動する。

CakePHP 2.0の見る順番

CakePHP 2.0では、URL、routes、Controller、Model、Viewの順に見る。
バグ修正でも機能追加でも、まずこの流れで関係ファイルを探す。

1. URL 例: /users/edit/1。URLからControllerとactionを推測する。
2. routes app/Config/routes.php。特殊なURL設定がないか見る。
3. Controller app/Controller/UsersController.php。edit() などのactionを見る。
4. Model app/Model/User.php。DB検索、保存、バリデーション、関連設定を見る。
5. View app/View/Users/edit.ctp。フォーム、表示、リンク、エラーメッセージを見る。

VS Codeで関係ファイルを探す

VS Codeでは Cmd + Shift + F で全体検索する。
ターミナルでは rg、なければ grep -R を使う。 

rg "UsersController"
rg "function edit"
rg "User->save"
rg "email"
rg "users"
grep -R "UsersController" app
grep -R "function edit" app
grep -R "User->save" app
grep -R "email" app
画面の文字で探す ボタン名、見出し、エラー文でViewを探す。
URLで探す Controller名、action名、routes設定を探す。
DB項目で探す email、status、user_idなどのカラム名でModel、Controller、Viewを探す。
保存処理で探す save、saveAll、validates を探す。

バグ修正の流れ

バグ修正は、原因を見つける前に直し始めない。
まず再現し、ログを見て、関係ファイルを絞り、最小限だけ直す。

1. 再現する どの画面で、何を入力し、何を押すと起きるか確認する。
2. ログを見る app/tmp/logs/error.log と必要に応じてApacheログを見る。
3. ファイルを絞る Controller、Model、View、routes、Configのどこが関係するか決める。
4. 小さく直す 関係ないリファクタリングを混ぜず、原因に関係する行だけ直す。
5. 同じ操作で確認する 最初にバグが出た操作をもう一度行い、直ったか確認する。

実践例: 編集画面で保存できない

例として、ユーザー編集画面で保存できない時の見る順番。
保存できない原因は、Controllerだけでなく、ModelのバリデーションやViewのフォーム名にもある。 

rg "function edit" app/Controller
rg "public \\$validate" app/Model
rg "Form->input" app/View/Users
tail -f app/tmp/logs/error.log
Controller UsersController::edit() で、request data、save、redirect、Session flashを見る。
Model User.php の $validate、関連設定、beforeSaveを見る。
View edit.ctp のFormHelper、input名、hidden id、エラー表示を見る。
確認 保存できるか、入力エラーが出るか、ログに新しいエラーが出ないかを見る。

機能追加の流れ

機能追加は、作るものを小さく分けてから触る。
画面、Controller、Model、DB、バリデーション、権限、確認方法を先に分ける。

画面 app/View のctp、Elements、Layoutsを確認する。
処理 app/Controller のactionを追加または変更する。
DB app/Model のfind、save、関連設定を確認する。
入力チェック CakePHP 2.0ではModelの $validate に書かれていることが多い。
確認方法 正常系、異常系、権限、ログ、既存画面への影響を確認する。

DBまわりの注意

DB変更は影響が大きい。
カラム追加、型変更、データ修正、本番データ削除は、必ず会社の手順を確認してから行う。

接続情報 app/Config/database.php を見る。ただしパスワードをチャットやPRに貼らない。
DB変更 SQLファイル、SchemaShell、社内マイグレーション運用のどれかを確認する。
データ修正 UPDATEやDELETEはレビューなしで実行しない。実行前にSELECTで対象件数を見る。

一時デバッグ

古いCakePHPやCentOS環境では、Xdebugをすぐ使えないことがある。
その場合はログ出力で値を見る。ただしコミット前に必ず消す。 

CakeLog::write('debug', print_r($this->request->data, true));
debug($this->request->data);

消し忘れ確認: VS Codeの全体検索で debug(、CakeLog::write、var_dump、print_r を探す。

差分確認

コミット前に、VS CodeのSource Controlで変更ファイルを1つずつ開く。
ターミナルでも git diff を見て、不要な変更がないか確認する。 

git status
git diff
消し忘れ debug、var_dump、print_r、一時コメント、不要な空行。
関係ない変更 触る予定のないCSS、設定ファイル、改行だけの変更。
危険な変更 database.php、SQL、共通Component、共通Layout。

確認とテスト

CakePHP 2.0では testsuite shell が使われる構成がある。
まず app/Console/cake で使えるコマンド一覧を確認する。 

app/Console/cake
app/Console/cake testsuite app Model/User
app/Console/cake testsuite app Controller/UsersController
正常系 通常の入力で期待通り保存、表示、遷移するか。
異常系 空欄、長すぎる文字、不正な形式でエラー表示されるか。
権限 ログイン前、一般ユーザー、管理者、自分以外のデータでどうなるか。
ログ 操作後に error.log やApacheログへ新しいエラーが出ていないか。

コミットとPR

関係あるファイルだけをステージする。
VS CodeのSource Controlでも、ターミナルでも確認できる。 

git add app/Controller/UsersController.php
git add app/Model/User.php
git add app/View/Users/edit.ctp
git commit -m "Fix user edit validation"
git push origin fix/ticket-123-user-edit-error
PRの概要 何のバグ、何の機能追加かを1行で書く。
変更内容 Controller、Model、View、DBなど、どこをどう変えたかを書く。
確認方法 URL、入力値、期待結果、ログ確認、テスト結果を書く。
